# Original Concept Context

이 문서는 현재 `peter-anchor-risk-modeling` repo가 어떤 팀 프로젝트 원본 기획에서 출발했는지 정리한다.

주의:

- 원본 보고서, 포스터, 상장 자료는 팀 경진대회 산출물이다.
- 이 문서는 원본 내용을 개인 단독 성과로 주장하기 위한 문서가 아니다.
- 현재 repo에서 개인 작업으로 강조할 범위는 원본 기획 중 AI 위험 감지 파트를 실제 공개 데이터 기반 모델링으로 재정의하고 검증한 후속 고도화 작업이다.

원본 자료 유형:

- 팀 실무보고서
- 팀 포스터
- 수상 증빙 자료
- 발표/기획 관련 원본 파일

위 원본 자료는 팀 산출물이며 개인정보, 팀원 기여, 대회 제출물 권리 이슈가 있을 수 있으므로 이 repo에 공개하지 않는다. 이 repo에는 공개 가능한 분석 결과와 데이터 기반 고도화 문서만 정리한다.

## 1. 원본 팀 기획의 핵심

원본 프로젝트명은 다음과 같다.

> 피터 앵커: 거미줄형 비상 그물 흡착망 AI 안전 웨어러블 로봇

원본 팀 기획의 핵심 문장은 다음과 같다.

> 기존 로프 작업은 유지하고, 위험 감지는 피터 앵커가 먼저 수행한다.

즉, 작업자를 완전히 대체하는 외벽 청소 로봇이 아니라, 기존 로프 작업을 유지하면서 작업자의 위험을 먼저 감지하고 비상 시 보호 장치를 작동시키는 안전 보조 시스템이다.

## 2. 원본 팀 기획의 주요 구성요소

원본 보고서 기준으로 Peter Anchor는 다음 구성요소를 가진다.

| 구성요소 | 원본 팀 기획에서의 역할 | 현재 repo에서의 처리 |
| --- | --- | --- |
| 헬멧/상체 IMU | 기울기, 흔들림, 급가속 감지 | 실제 공개 IMU 데이터 기반 모델링으로 구현 |
| 로프 장력 센서 | 장력 급감, 파단 전조 감지 | 공개 데이터가 없어 future sensor로 분리 |
| 스마트 하네스 | 체결 상태, 하중 분산 감지 | 공개 데이터가 없어 future sensor로 분리 |
| 허리 제어박스 | AI 판단, 배터리, 통신 모듈 | 안전 정책 레이어로 개념화 |
| 웹 카트리지 | 비상 안전망 수납 및 전개 준비 | emergency protection layer로 분리 |
| 고강도 섬유 안전망 | 충격 분산, 접촉 면적 확대 | 실제 전개/충격 데이터가 없어 검증 과제로 분리 |
| 작업자 경고 UI | LED, 음성, 진동 등 경고 | `safety_response_policy.py`의 response action으로 반영 |

## 3. 원본 팀 기획의 위험 감지 알고리즘

원본 보고서의 위험 감지 의사코드는 다음 입력을 가정했다.

```text
rope_tension
imu_acceleration
body_angle
harness_pressure
worker_sway
wind_speed
suction_ready
battery_level
```

위 입력을 조합해 다음 위험 유형을 판단하는 구조였다.

| 위험 유형 | 판단 근거 | 대응 |
| --- | --- | --- |
| 작업 전 점검 필요 | 장비 상태 이상 | 작업 시작 제한, 관리자 점검 요청 |
| 착용 상태 이상 | 하네스 체결 불량 | 작업자 경고, 체결 재점검 |
| 추락 전조 | 로프 장력 급감 + 급가속 + 비정상 각도 | 긴급 경고, 흡착 패드/안전망 준비 |
| 강풍 충돌 위험 | 강풍 + 흔들림 증가 | 작업 중지 권고, 자세 안정화 |
| 비상 안정화 | 흡착 가능 + 추락 전조 | 흡착 패드 전개, 안전망 전개, 구조 요청 |
| 정상 상태 | 위험 조건 없음 | 실시간 모니터링 유지 |

## 4. 현재 repo에서 개인 작업으로 정리하는 범위

현재 repo는 원본 팀 기획 전체를 구현한 것이 아니라, 그중 데이터로 검증 가능한 부분을 분리해 후속 고도화한 것이다.

현재 구현 범위:

```text
헬멧/상체 IMU
-> fall-like motion 감지
-> fall_risk / normal_activity 분류
-> 안전 대응 정책으로 연결
```

현재 제외한 범위:

```text
로프 장력 센서
하네스 압력 센서
흡착 패드 성능
안전망 전개 시간
충격 흡수 성능
근력 보조 actuator 제어
```

제외한 이유는 단순하다. 공개 실제 데이터가 없기 때문이다. 이 repo는 가상 데이터를 만들지 않고, 실제 공개 데이터로 검증 가능한 범위만 구현한다.

## 5. 원본 MVP와 현재 repo의 대응

원본 보고서의 MVP 단계는 “핵심 위험 감지 검증”이었다.

| 원본 단계 | 목표 | 현재 repo와의 관계 |
| --- | --- | --- |
| MVP | 장력/자세/가속도 감지 | IMU 기반 fall-risk 감지로 일부 구현 |
| 시제품 | 착용형 하네스, 제어박스, 흡착 안전망 | 안전 정책과 future architecture로 설계 |
| 장기 제품 | AI 판단, 자동 안정화, 관제 연동 | 현장 데이터 수집 이후 확장 과제 |

따라서 현재 repo는 원본 팀 기획의 MVP 중에서도 **데이터 기반 AI 위험 감지 검증**에 해당하는 부분을 후속으로 정리한 것이다.

## 6. 왜 현재 repo에 함께 두는 것이 좋은가

새 repo를 만들기보다 현재 repo 안에 원본 팀 기획 맥락을 함께 정리하는 편이 좋다. 다만 표현은 반드시 “팀 산출물 기반 후속 개인 고도화”로 유지해야 한다.

이유:

- 현재 AI 모델링이 원본 Peter Anchor 팀 기획에서 출발했다는 흐름이 분명해진다.
- 원본 보고서의 “AI 위험 점수화” 아이디어를 실제 공개 데이터 기반 모델링으로 후속 검증했다는 스토리가 생긴다.
- 포트폴리오에서 팀 프로젝트 배경과 개인 데이터 검증 작업을 구분해서 보여줄 수 있다.
- README가 제품 기획과 모델링 결과를 자연스럽게 연결한다.

원본 발표자료, 포스터, 상장 스캔본 자체는 공개하지 않는 것을 기본 원칙으로 둔다. 팀원 전원의 동의와 공개 범위 합의가 없는 한 public repo에 포함하지 않는다.

추천 구조:

```text
peter-anchor-risk-modeling/
  README.md
  docs/
    original_concept_context.md
    evidence_sources.md
    ai_modeling_process.md
    sequence_model_comparison.md
    system_architecture.md
    backpack_safety_net_integration.md
    wearable_robot_extension_plan.md
  src/
  data/
  outputs/
```

별도 repo가 필요한 경우:

- 팀 프로젝트 산출물과 개인 고도화 작업을 명확히 분리해야 할 때
- 팀원들과 함께 private archive를 관리하고 싶을 때
- public repo에는 개인 후속 모델링만 두고, 원본 자료는 private repo 또는 로컬 보관으로 제한하고 싶을 때

현재 단계의 결론:

> 이 repo는 Peter Anchor 팀 기획의 AI 위험 감지 파트를 실제 데이터 기반으로 검증한 후속 개인 고도화 repo로 유지하는 것이 가장 적절하다. 원본 팀 산출물은 public repo에 올리지 않고, 필요하면 팀원 동의하에 private archive로만 관리한다.
